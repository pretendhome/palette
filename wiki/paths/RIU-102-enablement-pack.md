#### ⚡ QUICK START (5 minutes)

**You'll produce**: A 3-part enablement starter: one quickstart guide, one working example, and one troubleshooting entry.

**Steps**:
1. Pick one system you've built that someone else needs to use.
2. Write a quickstart: the 5 steps to go from zero to "it works." No background, no theory — just steps.
3. Create one working example: a complete, copy-pasteable example that demonstrates the core use case.
4. Write one troubleshooting entry: the most common error, why it happens, and how to fix it.
5. Package as a clean enablement starter.

**⚠ Common mistakes** (watch for these):
- Quickstarts that assume knowledge the reader doesn't have ("first, configure your environment" — how?).
- Examples that are too simple to be useful or too complex to understand.
- Troubleshooting that says "check the logs" without saying what to look for.
- Writing for yourself instead of for the person who's never seen this system.

**✓ Check your work** — standard 4-point check with confidence delta.

---

#### 🔨 APPLIED (15-30 minutes)

**You'll produce**: A work-ready enablement pack with quickstart, concept guide, 3 examples, troubleshooting guide, and FAQ.

**Steps**:
1. Choose a real system. Identify your audience: what do they know? What don't they know?
2. Write the quickstart: zero to working in under 10 minutes. Test it by following your own steps on a clean setup.
3. Write a concept guide: the 3-5 key concepts someone needs to understand (not everything — just enough to be productive).
4. Create 3 working examples: basic use case, common variation, and edge case handling.
5. Build a troubleshooting guide: top 5 errors with symptoms, causes, and fixes.
6. Write an FAQ: the 5 questions everyone asks. Answer them before they ask.
7. Test: give the pack to someone unfamiliar with the system. Can they get started without asking you?

**✓ Check your work** — standard 6-point check with confidence delta.

---

#### 🏗️ PRODUCTION (30-60 minutes)

**You'll produce**: A portfolio-grade enablement system with tiered documentation, interactive examples, feedback loops, and maintenance plan.

**Steps**:
1. Tier your documentation: quickstart (5 min), tutorial (30 min), reference (complete), troubleshooting (searchable).
2. Create interactive examples: not just code to read, but code to run with expected outputs.
3. Build a feedback loop: how do users report problems with the docs? How do you track which docs are most/least useful?
4. Design a maintenance plan: who updates docs when the system changes? How do you detect stale documentation?
5. Add onboarding paths: different paths for different audiences (developer, operator, business user).
6. Create a contribution guide: how can users improve the docs themselves?
7. Document the enablement system itself (meta-documentation).

**✓ Portfolio review** — standard 7-point review with confidence delta.
