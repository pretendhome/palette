# How do I challenge whether I am building the right thing before committing to implementation?

Use a structured premise challenge before any implementation begins. Three questions: (1) Is this the right problem to solve, or could a different framing yield a simpler or more impactful solution? (2) What is the actual user or business outcome — is this plan the most direct path, or is it solving a proxy problem? (3) What would happen if we did nothing — is this a real pain point or a hypothetical one? Then map existing code leverage (what already partially solves this?), produce a dream state delta (current state → this plan → 12-month ideal), and select a scope mode: expansion (dream big, opt-in per item), selective expansion (hold scope but surface opportunities), hold scope (make it bulletproof), or reduction (strip to essentials). Every scope change requires explicit human opt-in. This prevents the most expensive engineering failure: building the wrong thing well.
