# How do I create an internal demo narrative that achieves alignment without overpromising?

RIU-290 structures the demo narrative in four acts. Act 1 — Problem (2-3 minutes): start with the customer/user pain point, not the technology. Use a specific scenario that the audience recognizes. Example: "Today, when a customer calls about a return, the agent spends 4 minutes searching three systems. The customer waits. The agent is frustrated. 30% of these calls escalate unnecessarily." Act 2 — Value (5-7 minutes): demonstrate the solution using the exact scenario from Act 1. Show the live system (never a slideshow). Narrate what is happening: "The agent asks the customer for their order number. The system retrieves the order, checks return eligibility, and suggests a resolution — all in 8 seconds." Show the measurable improvement: "4 minutes reduced to 8 seconds. Escalation rate dropped from 30% to 12% in pilot." Act 3 — Limits (2-3 minutes): this is where most demos fail by omission. Explicitly state what the system cannot do, what edge cases are not handled, and what assumptions it relies on. Example: "This works for standard returns. Custom orders, cross-border returns, and warranty claims are not yet supported. The system requires the order to be in our database — marketplace orders are not covered." Act 4 — Next Decisions (2-3 minutes): present 2-3 decisions that the audience needs to make. Example: "Should we expand to custom orders next, or focus on reducing the escalation rate further? Do we have budget for Phase 2?" Common failure: demo-driven hype where the audience leaves believing the system does more than it actually does. The Limits section prevents this.
