# How do I implement ranking explanation rules that provide deterministic reason codes instead of LLM-generated prose?

RIU-034 requires deterministic reason codes attached to every ranked output. Design principles: (1) Finite Code Set: define a closed set of reason codes (typically 15-40) that cover all possible ranking factors. Example codes: RECENCY_BOOST, RELEVANCE_HIGH, PRICE_COMPETITIVE, LOW_STOCK_PENALTY, USER_PREFERENCE_MATCH, GEOGRAPHIC_PROXIMITY. Each code has a human-readable description and a polarity (positive/negative). (2) Signal-to-Code Mapping: each ranking signal maps to exactly one reason code. If a signal contributes positively, attach its reason code with a positive weight. If negatively, attach with a negative weight. The top 3 reason codes by absolute weight become the explanation for that result. (3) Deterministic: given the same inputs, the same reason codes must be produced. This means no LLM in the explanation path — reason codes are computed from the ranking signals, not generated by a language model. LLM-generated explanations are unreliable because they can fabricate plausible-sounding but incorrect reasons. (4) Debugging: reason codes make ranking changes debuggable. When a result drops in rank, compare its reason codes before and after to identify which signal changed. This replaces "the model changed its mind" with "the RECENCY_BOOST signal expired." Cross-reference with RIU-027 (Deterministic Scoring) for the ranking signal computation itself.
