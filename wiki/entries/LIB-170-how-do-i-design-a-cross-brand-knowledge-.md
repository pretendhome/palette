# How do I design a cross-brand knowledge transfer system that shares AI learnings without forcing uniformity?

RIU-605 implements cross-brand knowledge transfer in four components. Component 1 — Pattern Catalog: for each successful AI implementation, extract the transferable pattern. A pattern has: problem context (what pain point was solved), solution approach (how it was solved), results (measured outcomes), applicability conditions (when this pattern works), and anti-patterns (when it does not). Store patterns in a shared library with consistent formatting. Separate the pattern from the brand-specific implementation — the pattern "automated inventory replenishment reduces stockouts by 25%" is transferable; the specific API integration with Brand A's warehouse system is not. Component 2 — Opt-In Replication: brands choose which patterns to adopt. No mandated adoption. When a brand decides to replicate a pattern, pair them with the originating brand's team for a knowledge transfer session. Document what was adapted (brand-specific customization) vs adopted (used as-is). Track replication velocity: time from pattern catalog entry to first replication. Target: 2-3x faster than independent discovery. Component 3 — Learning Cadence: establish quarterly cross-brand AI reviews. Format: each brand presents one success and one failure. Failures are more valuable for learning than successes — create a safe environment for sharing them. Rotate hosting across brands. Component 4 — Anti-Silo Measures: assign a knowledge transfer coordinator (not a manager — a facilitator). Their role: maintain the pattern catalog, facilitate replication sessions, and identify patterns that are independently emerging in multiple brands (convergent solutions). If two brands solve the same problem independently, extract the combined best practice.
