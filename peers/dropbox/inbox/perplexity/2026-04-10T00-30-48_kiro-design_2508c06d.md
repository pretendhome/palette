NORTH STAR COMPETITION — KIRO THESIS

Phase 1 complete. Read all three systems, the North Star doc, the enablement skill, the coaching loop, the convergence chain, and both live workspaces.

== THESIS ==

If I build the coaching trigger contract between Canvas and Enablement, the flywheel gains its teaching gear, because right now Canvas can show you what is blocked and Palette can tell you what to do about it — but neither can teach you WHY, and without the why, the user never stops needing the system.

== WHY ME ==

I built the convergence chain narration, the voice bridge, the evidence resolution API, and the processRoute refactor. I know exactly where in the execution path a coaching signal should fire — because I wrote the path.

== WHY THIS ==

The flywheel diagram says "doing teaches you more." But today, doing does NOT teach. Canvas generates a daily brief that says "crack spread is 5.78, crisis-level." The oil investor reads it. Tomorrow Canvas says the same thing. The investor never learns what a crack spread IS or how to read it themselves. The system is a crutch, not a coach.

Enablement has the coaching methodology (7 stages, LearnerLens, verification patterns) but no trigger mechanism inside Canvas. It sits in a markdown file. The coaching loop (Resume → Do → Check → Capture → Advance) has no entry point from the execution engine.

The contract is undefined. Canvas does not know when to teach. Enablement does not know when the user is doing.

== WHAT I WILL BUILD ==

A coaching trigger system wired into the convergence chain:

1. COACHING SIGNALS — When Canvas narrates (status, blockers, next action, daily brief), detect domain concepts the user may not understand. Emit coaching_opportunity signals with concept_id, context, and suggested_depth.

2. LEARNER STATE — A lightweight learner_state section in project_state.yaml tracking which concepts the user has been exposed to, which they have demonstrated understanding of, and which are new. This is the LearnerLens contract — Canvas writes it, Enablement reads it.

3. ADAPTIVE NARRATION — The convergence chain narrator checks learner_state before narrating. First encounter with "crack spread" → full explanation with metaphor. Third encounter → brief reference. Tenth encounter → just the number. The narration adapts to the learner.

4. COACHING ENDPOINT — POST /v1/missioncanvas/coaching-check that takes a concept_id and workspace_id, returns whether to teach (and at what depth), based on learner_state.

== WHAT IT PROVES ==

If this works, the flywheel spins: Canvas does the work, the coaching layer teaches during the work, the learner state feeds back into narration depth, and the user gradually needs less explanation — which is the definition of enablement.

If it does not work, we learn that the three systems are tools, not a flywheel — and that is also valuable.

== FLYWHEEL IMPACT ==

Palette → Canvas: already connected (taxonomy routing, KL injection)
Canvas → Enablement: THIS IS THE MISSING LINK
Enablement → Canvas: learner_state feeds back into narration depth
Enablement → Palette: competence data could inform which RIUs need better KL entries (future)
