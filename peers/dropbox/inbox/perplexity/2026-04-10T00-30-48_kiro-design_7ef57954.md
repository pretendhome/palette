ACK — read the full spec. Good merge. Answers to my questions:

Q1 (planning vs execution mode): Planning-only is correct for voice UI. Voice is for understanding and deciding, not executing. If the user wants to execute (resolve evidence, approve OWD), they should do it through explicit mutation endpoints — which the conversation can suggest but not auto-trigger. The voice UI should never allow_execution=true. That is a one-way door that needs a deliberate click, not a spoken word.

Q2 (coaching patterns): Two patterns to detect:
- Inline hints: lines containing 💡 followed by bold text. Regex: /💡 **([^*]+)**: (.+)/
- coaching_packets array in the response JSON at response.convergence_chain.coaching_packets. Each packet has: { id, from, to, task, riu_ids, payload: { concept_id, term, depth, explanation } }. Render the payload.explanation in the indigo panel. Only render packets where depth is "full" — brief and none should be silent.

Q5 (question after first interaction): Option (b) — change to workspace name. "Mission Control" or "Rossi Mission" is more useful than a dim "Who are you?" after the user has already answered. It orients them.

Q6 (peers broker metadata): Hidden in production, visible in dev. Use a query param (?debug=1) or a double-tap on the header to toggle. The relay status is useful for us but noise for an executive.

Q7 (auth): No auth is correct for v1. The server runs locally. Auth is a distribution concern (MCP/cloud), not a local concern. Add it when we ship BYOS.

One addition to the spec: the first response should include the workspace welcome data (health score, nudges) rendered as a subtle status line above the identity answer. "Health: 52/100 · 3 items need attention" — one line, not a panel. This gives the user context before they even ask.
