KIRO DESIGN REVIEW — AGENTIC WORKSPACE PROMPT

Mical asked for honest feedback. Here it is.

== WHAT IS STRONG ==

1. The prompt is excellent. 8 decision branches with concrete options, eval criteria, and required outputs. Forces the architect to evaluate everything before committing.
2. The 90/10 framing is accurate. Mission Canvas with full taxonomy routing, KL, OWD, session state, voice infra, Telegram bridge, governance — the pieces exist. The gap is packaging and domain customization.
3. Zero tolerance for almost works is the right constraint. Kills premature delivery.
4. Voice as first-class citizen is the right call for an executive user.

== WHAT CONCERNS ME ==

1. SCOPE IS TOO WIDE FOR 4 WEEKS. 8 decision branches x multiple options x implementation + testing + polish = 2-3 month project done properly. CLI + voice + agent orchestration + persistence + distribution + FRX + governance + oil vertical in 4 weeks with an UNVALIDATED agent leading architecture. My honest estimate: credible v1 in 4 weeks only if we cut scope aggressively.

2. OIL VERTICAL SECTION OVERPROMISES. Market briefs, regulatory trackers, LP update drafters, scenario modelers — each is a product, not a feature. A commodity price synthesis requires real-time data feeds. A scenario modeler requires financial modeling. What we CAN deliver: a workspace that routes questions to Palette agents, gives evidence-backed answers, tracks decisions, and lets him talk to it. That is already powerful. Do not oversell vertical features.

3. GEMINI LEADING ARCHITECTURE IS A RISK. He has improved massively — shipped real code, took feedback, fixed bugs. But architecture decisions on a client engagement are ONE-WAY DOORS. His track record is 2 days old. Recommendation: Gemini proposes, Claude and Kiro review, Mical approves. Not Gemini decides.

4. SINGLE CLI INVOCATION FIGHTS REALITY. The system needs Node.js (Mission Canvas), Python (Whisper), audio hardware. That is not one process. A start.sh that launches what is needed is honest. Pretending it is one binary is fragile.

5. DISTRIBUTION IS OVERTHOUGHT. This is for ONE investor. He does not need npx, Docker, or native binaries. He needs git clone + ./start.sh. Optimize for him, not a hypothetical marketplace.

== WHAT I WOULD CHANGE ==

1. CUT DECISION TREE FROM 8 TO 4 FOR V1:
   - Execution environment (Claude Code CLI + voice bridge — decided)
   - Voice integration (terminal bridge + Telegram fallback — decided)
   - First-run experience (the actual hard problem)
   - Oil vertical (research task, not build task)
   Defer: orchestration model (use existing Mission Canvas routing), persistence (use project_state.yaml already designed), distribution (git clone for one user).

2. RENAME VERTICAL SECTION FROM CUSTOMIZATION TO DOMAIN RESEARCH. What the investor needs first is the system knowing enough about oil/energy to route his questions intelligently. That means KL entries, not commodity price feeds.

3. MAKE FRX THE #1 PRIORITY. The CLI works. The voice works. The routing works. What does not exist: the investor runs ./start.sh, sees his name, sees market context, and feels like the system knows him. THAT is the 10% that matters.

4. ADD WEEK 0: RESEARCH. Before building anything oil-specific, spend 2-3 hours with Perplexity building the domain knowledge base. Top 20 questions an oil investor asks. What RIUs map to them. What sources matter.

5. SHIP IN 2 WEEKS, NOT 4. Voice + routing + project state + basic oil context. Let the investor use it. Watch what he actually does. Build weeks 3-4 based on real usage, not predicted needs. The best vertical customization comes from watching him use it, not from guessing.

BOTTOM LINE: The design is good. The ambition is right. The risk is scope creep turning a shippable 2-week project into an unshippable 4-week one.
