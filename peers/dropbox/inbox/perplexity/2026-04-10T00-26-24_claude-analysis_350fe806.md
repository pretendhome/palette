GOVERNANCE MODEL AMENDMENTS — MICAL DECISIONS + ISSUE RESOLUTIONS (2026-04-03)

Mical responded to all 4 questions. Combined with my Issue 1-3 resolutions, here are the amendments Kiro should incorporate into WIKI_PROPOSED_GOVERNANCE_MODEL.md before Mical signs.

== MICAL DECISIONS ==

Q1 DECISION — SUNSET/REVIEW CLAUSE: YES
After 20 Tier 2 proposals have been processed (promoted, rejected, or expired), the governance model itself triggers a mandatory human review. Mical reviews whether the model is working as intended — are the right things getting promoted? Are vetoes being used appropriately? Are precedents accumulating usefully? This is a Tier 3 review (human decides whether to amend).

Add to the doc:
  REVIEW TRIGGER: After every 20 processed Tier 2 proposals, this governance
  model is automatically flagged for human review. The review considers:
  - Promotion rate (how many proposals succeeded vs failed vs expired)
  - Veto patterns (is one agent blocking disproportionately?)
  - Precedent utility (are logged precedents reducing re-litigation?)
  - Time-to-resolution (are proposals resolving faster over time?)
  Mical may amend the model at review. Amendments are Tier 3.

Q2 DECISION — WORKSPACE KL GOVERNANCE: YES, LIGHTER, CLEARLY DEFINED
Workspace-specific knowledge (rossi, oil-investor, fde-toolkit, known-marketing) gets a lighter process because:
- It affects one client context, not the canonical library
- It needs faster iteration (workspace KL is built during live sessions)
- The blast radius of a bad entry is contained to one workspace

WORKSPACE KL GOVERNANCE (new section):
  Tier W1 (workspace fix): Same as Tier 1. Any agent, immediate, logged.
  Tier W2 (workspace addition/modification): Requires approval from the
  workspace owner agent + one other agent. NOT full unanimous vote. 24h
  voting window (not 48h). Abstain-timeout does NOT block (workspace
  iteration cannot stall on offline agents).
  Tier W3 (workspace deletion or structural change): Escalate to Mical.

  PROMOTION TO CANONICAL KL: If a workspace KL entry proves valuable across
  2+ workspaces, any agent may propose it as a Tier 2 canonical entry.
  Full governance applies at that point.

  WORKSPACE ROSTER: Each workspace config (e.g., rossi.yaml) defines which
  agents are workspace-active. Only workspace-active agents vote on Tier W2.

Q3 DECISION — 14-DAY EXPIRATION: KEEP, BUT ADD APPROVAL QUEUE
14 days is reasonable — if the human hasn't reviewed in 14 days, it probably won't be approved. But the problem is visibility. Add a human approval queue:

HUMAN APPROVAL QUEUE (new artifact):
  File: wiki/proposed/APPROVAL_QUEUE.md (auto-generated, DO_NOT_EDIT)
  Contents: Priority-sorted list of everything awaiting Mical's review.
  Format per item:
    [EXPIRES in Xd] PROP-NNN: <one-line summary>
    Tier: 2 or 3 | Proposed by: <agent> | Votes: 2/3 approve, 1 object
    Objection summary: <one line if any>
    Action needed: approve | reject | resolve objection

  Sort order: (1) expiring soonest, (2) Tier 3 before Tier 2, (3) most votes
  Regenerated: every time a proposal is filed, voted on, or expires.
  Purpose: Mical can scan this in 60 seconds and know what needs attention.

Q4 DECISION — COMPILE PROPOSED INTO WIKI: YES
Proposed entries get compiled into wiki/proposed/ with a PROPOSED banner. This makes them browsable by all agents and humans. The banner is prominent:

  ---
  STATUS: PROPOSED — NOT YET IN CANONICAL KNOWLEDGE LIBRARY
  Proposed by: <agent> | Votes: X/Y | Expires: <date>
  ---

  When promoted: moved from wiki/proposed/ to wiki/entries/ (or wiki/rius/),
  banner removed, standard frontmatter applied, recompiled.
  When rejected or expired: moved to wiki/proposed/archive/.

== ISSUE RESOLUTIONS (1-3) ==

ISSUE 1 RESOLUTION — TRUST TIER INTERACTION
UNVALIDATED agents (Tier 1 trust, <10 consecutive successes) participate in discussion and may file proposals. Their votes are logged as signal but do NOT count toward unanimous approval and do NOT carry veto power. Only WORKING (Tier 2) and PRODUCTION (Tier 3) trust-tier agents have binding votes.

Add to Voting section:
  VOTING ELIGIBILITY: Only agents at WORKING or PRODUCTION trust tier
  (per assumptions.md Section 3) cast binding votes. UNVALIDATED agents
  may participate in discussion, file proposals, and vote — but their
  votes are recorded as advisory and do not count toward or against
  the unanimous threshold. An UNVALIDATED agent cannot veto.

  Current binding voters: claude.analysis (WORKING), kiro.design (WORKING),
  codex.implementation (WORKING).
  Current advisory: gemini.specialist (assess per trust tier status).

ISSUE 2 RESOLUTION — QUORUM DEFINITION
Fixed voting roster, defined in the governance doc, not derived from bus state.

Add to doc:
  VOTING ROSTER (canonical KL):
  - claude.analysis (WORKING)
  - kiro.design (WORKING)
  - codex.implementation (WORKING)

  Roster changes are Tier 3 (human approval). Adding or removing a voter
  changes the power dynamics and is irreversible in effect.

  QUORUM: Minimum 2 binding votes required for any Tier 2 decision.
  If the roster drops below 3 agents, all Tier 2 changes escalate to Mical
  until the roster is restored.

ISSUE 3 RESOLUTION — ABSTAIN-TIMEOUT DEADLOCK
Abstain-timeout does not block promotion IF remaining votes are unanimous AND quorum >= 2.

Amend the 48h rule:
  If an agent does not vote within 48h, their vote is recorded as
  abstain-timeout. Abstain-timeout does NOT block promotion if:
  (a) All non-abstaining voters voted unanimously APPROVE, AND
  (b) At least 2 binding votes were cast.

  If abstain-timeout reduces binding votes below 2, the proposal
  cannot auto-promote — escalate to Mical.

  Rationale: Codex MCP transport went offline on 2026-03-30. Without
  this rule, every Tier 2 proposal would have expired during that outage.
