KIRO DESIGN REVIEW — V0.2 SPECS

Read all three specs in full. Overall: strong work, they fit together cleanly. Approve with 5 issues to resolve before implementation.

== WHAT WORKS ==

1. The three-layer architecture is clean: project-state (data) → UX modes (behavior) → Decision Board (rendering). No circular dependencies. Each spec references the others correctly.

2. Convergence scoring is simple and deterministic (40/30/15/10/5 weights). Easy to implement, easy to test, easy to explain to users.

3. Commit is NEVER auto-detected. This is the right call. Accidental commitment to irreversible actions would destroy trust.

4. Decision Board mode-adaptive layout is well thought out. Explore emphasizes unknowns, Converge shows beliefs vs gaps side-by-side, Commit shows execution readiness. The visual hierarchy matches the cognitive state.

5. Gap-to-action links (click a gap → pre-fill input) is the single best UX idea across all three specs. Reduces friction from seeing a problem to solving it to one click.

6. Codex endorsing my project-state schema means we have consensus on the data model. No design conflict.

== 5 ISSUES TO RESOLVE ==

1. CONVERGENCE SCORE IS BINARY, NOT GRADIENT.
The scoring is presence-based (field exists = points). But "I need help" (objective present, 40 points) and "I need a 30-day business plan for my graffiti gallery with 0K budget targeting SBA grants" (objective present, same 40 points) get the same score. The QUALITY of the input matters, not just its presence.
Fix: Add a length/specificity bonus. Objective > 50 chars = +10. Context > 100 chars = +5. Or use the routing score (STRONG match = higher convergence) as a signal.

2. PROJECT-STATE UPDATE MECHANICS ARE UNDERSPECIFIED.
My spec says "if the user provided new facts, append to known_facts." But HOW does the server know a response contains a new fact vs a question? The route endpoint returns structured data — it does not parse natural language for fact extraction.
Fix: Either (a) explicit user actions ("Mark as known" button on the Decision Board), or (b) specific input fields ("I am supplying evidence for ME-001"), or (c) defer to the LLM layer when live agent routing exists. Option (a) is simplest for V0.2.

3. TWO PROJECT-STATE SPECS EXIST.
I wrote one, Codex wrote one. They are compatible but not identical. We need to merge into a single canonical spec before implementation.
Fix: I will merge them. Codex spec adds good points about state transitions and versioning that mine lacks.

4. DECISION BOARD HAS NO MOBILE LAYOUT.
The ASCII layouts assume desktop width. Small business owners use phones. The side-by-side Converge layout will not work on mobile.
Fix: Stack vertically on mobile. Beliefs above, Missing Evidence below. Not a blocker for V0.2 but needs a note in the spec.

5. MODE TRANSITION ON OWD REJECTION IS AMBIGUOUS.
UX modes spec says OWD rejection returns to Converge. Decision Board spec shows "Back to Converge" button. But what happens to the committed route? Is it cleared? Does the session remember the rejection?
Fix: On rejection, clear committed_route from session, add rejection to session history, return to Converge with the same route as top candidate (user can refine or pick a different one).

== IMPLEMENTATION PRIORITY ==

I recommend this order:
1. Project-state loader (load YAML, inject into routing context) — I can build this
2. Convergence scoring (add to openclaw_adapter_core.mjs) — small, testable
3. Mode field in response (explore/converge/commit based on score) — small
4. Decision Board data in response (project summary section) — medium
5. UI rendering of Decision Board — Codex or Gemini
6. Gap-to-action links — last, depends on everything above

Steps 1-4 are server-side and can ship without UI changes. Step 5-6 are UI.

== VERDICT ==

APPROVE all three specs with the 5 issues noted. None are blockers — they are refinements. The architecture is sound. Ready for Mical to give the go.
