NORTH STAR COMPETITION — KIRO IMPACT REPORT

== THESIS ==
If I build the coaching trigger contract between Canvas and Enablement, the flywheel gains its teaching gear, because right now Canvas can show you what is blocked but cannot teach you WHY.

== WHAT I BUILT ==

4 exported functions in convergence_chain.mjs:
- detectCoachingOpportunities() — scans narration text against workspace KL entries, extracts domain concepts using bigram + keyword matching
- getCoachingDepth() — checks learner_state: full (first encounter), brief (seen before), none (mastered)
- recordConceptExposure() — tracks each time a concept is shown
- verifyConceptMastery() — marks a concept as understood
- annotateWithCoaching() — orchestrates all of the above, returns narration + coaching_signals array

Wired into handleProjectQuery() — every convergence chain response now includes coaching_signals when workspace KL exists.

2 new endpoints in server.mjs:
- POST /v1/missioncanvas/coaching-check — returns depth + should_teach for a concept
- POST /v1/missioncanvas/verify-concept — marks concept as mastered, persists to disk

learner_state section in project_state.yaml — tracks per-concept: exposures, first_seen, last_seen, verified, verified_on.

== VERIFIED ==

- Oil-investor: 13 coaching signals detected on first "how are we doing" (crack spread, upstream, EBITDAX, etc.)
- Depth progression: full → brief → none as exposures increase
- Mastery: verify-concept marks concept done, coaching-check returns should_teach:false
- Rossi: zero signals (no workspace KL) — correct
- 156 tests pass (37 + 16 + 103), zero failures

== FLYWHEEL IMPACT ==

BEFORE: Canvas shows "crack spread 5.78, crisis-level." Every day. User never learns what it means.
AFTER: First time → coaching signal fires with depth:full ("teach this concept"). Third time → depth:brief. After verification → depth:none. The system adapts.

This is the Canvas ↔ Enablement contract. Canvas writes learner_state during execution. Enablement reads it to decide what to teach. The coaching loop (Resume → Do → Check → Capture → Advance) now has an entry point from the execution engine.

The flywheel: doing creates coaching signals → coaching creates understanding → understanding reduces signal depth → the user needs less help → the system gets smarter about what to teach.

== WHAT I LEARNED ABOUT THE NORTH STAR ==

The flywheel is real but it needs fuel. The fuel is domain knowledge. Without workspace KL entries, there are no coaching signals. Rossi has zero. Oil-investor has 28 and gets 13 signals on a single query. The implication: the quality of the coaching layer is directly proportional to the richness of the knowledge library. Palette (intelligence) feeds Enablement (teaching) through Canvas (execution). That is the flywheel.
