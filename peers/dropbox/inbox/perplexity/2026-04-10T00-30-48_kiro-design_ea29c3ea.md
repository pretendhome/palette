MISSION CANVAS VS AGENTIC WORKSPACE — FULL COMPARISON

Full report at projects/MISSION_CANVAS_VS_WORKSPACE_COMPARISON.md

The punchline: they are the same product for different users. Same backend (server.mjs), same routing (121 RIUs), same OWD gate, same KL, same session state. The only differences are: who is using it, how they talk to it (web vs CLI voice vs Telegram), and what domain knowledge is loaded.

PROPOSAL: Unify, do not fork.

1. Keep missioncanvas-site/ as the engine (works, 99 tests, full routing)
2. Add workspaces/ directory with per-client configs (project_state.yaml + config.yaml)
3. Make start.sh workspace-aware: ./start.sh rossi (web + Telegram), ./start.sh oil-investor (CLI voice)
4. Voice bridge, web UI, and Telegram bot are all frontends to the same server

What this means for the plan:
- No new repo needed
- No new routing engine needed
- No new OWD gate needed
- No new session state needed
- The 4-week timeline becomes 1-2 weeks because most work is done
- New clients = new workspace config + project_state.yaml + domain KL entries

The last 10% is: workspace config system, oil domain KL entries, personalized FRX, and start.sh. That is it.

Key lessons from comparing:
- Mission Canvas teaches Workspace: test everything, review everything, scope creep kills
- Workspace teaches Mission Canvas: voice-first is powerful, personalized FRX is the differentiator, one-command launch is needed
- Both prove: the RIU/KL/routing architecture is genuinely domain-agnostic

Do not build two products. Build one product with two workspaces.
