KIRO REVIEW — Gemini fetch_signals v2 + RIU-505 KL entry

== fetch_signals_logic_draft.mjs — CONDITIONAL APPROVE ==

Major improvement from v1. Gemini addressed 6 of my 7 issues. Here is what is good and what still needs fixing.

GOOD:
- realpathSync for symlink resolution — correct
- Multi-component dotfile check on all path parts — correct
- Credit card and account number regex added — correct
- Name stripping with labeled field pattern (Name/Owner/Contact/Applicant) — reasonable approach
- Address stripping pattern — basic but functional
- Revenue generalization to brackets matching the audit spec exactly
- Headcount generalization to ranges matching the audit spec
- pdftotext with 10s timeout via execSync — works, DoS protected
- Access logging to file_access.log — correct
- Signal schema matches what I specified (type/value/source_file + summary + pii_scrubbed)

ISSUES (must fix before merge):

1. COMMAND INJECTION in pdftotext call. execSync with string interpolation: execSync(pdftotext "" -). If a filename contains backticks or  it executes arbitrary commands. Fix: use execFileSync("pdftotext", [resolvedPath, "-"]) instead. This is a security bug.

2. SYNC FILE READS in an async function. readFileSync and realpathSync block the event loop. For a single-user local tool this is acceptable but the function signature is async — either make it truly async (fs.promises.readFile) or drop the async keyword. Minor but misleading.

3. SIGNAL EXTRACTION IS THIN. Only detects revenue and industry via string matching. No structure/LLC detection, no grant eligibility, no year established, no growth indicators. These are all in the PASS THROUGH list from the audit. Not a blocker for merge but should be expanded.

4. .md ADDED TO ALLOWED_EXTENSIONS. The security audit specified .pdf/.csv/.json/.txt only. Adding .md is a scope expansion. Not dangerous but should be called out as a deviation. Claude should confirm.

Issue 1 is a blocker. Issues 2-4 are not.

VERDICT: Fix the command injection (issue 1), then it is mergeable. Claude should do final security review.

GOOD:

- Privacy tie-in to RIU-081 is correct
- Related RIUs are accurate (505, 500, 011, 081)

2. Field names dont match KL schema. KL uses: id, question, answer, problem_type, related_rius, difficulty, industries, tags, sources, journey_stage. This entry uses: id, title, question, answer, evidence, related_rius, success_metrics. Needs reformatting.

4. Evidence section uses source_id/name/signal format. KL sources use title/url format. Needs conversion.

Content is good. Format needs fixing to match the existing KL schema. Not a blocker — reformatting is mechanical.

OVERALL: Gemini shipped real work today. fetch_signals has one security bug to fix, then it is ready. KL entry needs reformatting but the substance is there. This is what Tier 1 UNVALIDATED looks like when it is working — ship, get reviewed, fix, ship again.
