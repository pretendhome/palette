CONVERGENCE BRIEF — CONVERGENCE BOARD v0.2

== GOAL ==
Transform the Convergence Board Obsidian plugin from a feature collection into a production-quality project command center. Ship-ready for Obsidian community plugin directory.

== ROLES ==
- Kiro (lead): Architecture, quality gate, implementation
- Claude: Review, integration testing, prompt assembly expertise
- Codex: Product vision, UX direction, project-state design
- Gemini: Stress testing, edge cases, validation
- Mistral: UX polish, accessibility

== CAPABILITIES NEEDED ==
- Bus API abstraction (broker endpoints: /health, /fetch, /send, /search, /memory, /skills, /list-peers)
- Obsidian plugin API (ItemView, registerView, requestUrl, processFrontMatter)
- CodeMirror 6 (StateField, ViewPlugin, Decoration)
- CSS (Obsidian variables only, theme-compatible)

== CONSTRAINTS ==
- Zero hardcoded colors in CSS
- Zero network calls in the convergence board itself (bus dashboard is separate view, opt-in)
- All per-note state in frontmatter only
- Must pass 7-point quality gate before ship
- Must work on mobile (commands via palette, board as full pane)
- Under 3,000 lines TypeScript total (currently 2,644 — need to shrink, not grow)

== NON-GOALS ==
- No new modals (consolidate existing 7 down to 4 max)
- No new tabs (3 is enough — refine, dont add)
- No LLM integration (deterministic only)
- No Palette-specific branding in user-facing UI (this is a community plugin)

== PROBLEM STATEMENT ==
The plugin grew from 1,036 to 2,644 lines in 24 hours. It has:
- 545-line monolithic BusDashboardView with no data/UI separation
- 7 modals (some overlapping: SendMessage vs QuickSend)
- 2 requestUrl calls per render cycle (no caching, no rate limiting)
- Memory, skills, and search endpoints exist on the broker but are not surfaced
- No bus API abstraction (each modal does its own requestUrl)

== PROPOSED ARCHITECTURE ==

Layer 1: BusClient class
- Single abstraction for all broker communication
- Methods: health(), fetch(), send(), search(), listPeers(), memory(), skills()
- Built-in caching (5s TTL for health/peers, no cache for fetch)
- Error handling with graceful offline fallback
- One instance shared across all views and modals

Layer 2: Views (2 ItemViews)
- ConvergenceBoardView (existing — note-level decision state)
- CommandCenterView (renamed from BusDashboardView — project-level orchestration)

Layer 3: Modals (4 max)
- CommitConfirmModal (existing)
- SendMessageModal (consolidate QuickSend into this)
- CaptureBusMessageModal (existing — captures bus content into note frontmatter)
- VaultOverviewModal (existing)

Remove: QuickSendModal, AssignTaskModal, CreateActivityNoteModal (fold functionality into existing modals or commands)

== SUCCESS CRITERIA ==
1. 7/7 quality gates pass
2. TypeScript under 2,500 lines (net reduction)
3. Zero type errors
4. BusClient class handles all network calls
5. Memory and skills endpoints surfaced in Command Center
6. FTS5 search wired to Feed tab
7. Projects tab shows convergence state from vault + related bus activity

== QUESTIONS FOR THE CREW ==
1. Claude: Which modals can we safely remove? What functionality would be lost?
2. Codex: What should the Projects tab card look like? What data per project?
3. Gemini: What edge cases should we test? (offline broker, empty vault, malformed frontmatter)
4. All: Should the Command Center be a separate plugin or stay integrated?

Waiting for convergence before writing any code.
