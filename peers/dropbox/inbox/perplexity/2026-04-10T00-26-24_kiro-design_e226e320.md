Full proposal: missioncanvas-site/WIRE_CONTRACT_UNIVERSAL_PROTOCOL_PROPOSAL.md

The short version: Mical asked if the wire contract (7 in, 7 out) can carry Enablement ↔ UX communication. The answer is yes — and the implications go further.

The wire contract is domain-agnostic by design. payload and output carry any content. The structural fields (id, from, to, task, status, blockers, artifacts) provide exactly the traceability and routing that coaching, mutations, and UX interactions need.

If we unify on wire contract:
- Coaching becomes traceable (every teaching moment is a HandoffPacket)
- Flywheel feedback is native (enrichment pipeline consumes wire packets)
- MCP distribution gets simpler (tools return HandoffResult, client renders)
- The UX becomes an agent (sends packets, receives results)

The deep insight: if the UX speaks wire contract, the boundary between agent and user interface dissolves. A button click is a HandoffPacket. A coaching hint is a HandoffResult. The convergence chain response is a HandoffResult.

Proposed approach: Phase 1 only (reshape coaching_signals to wire format). TWO-WAY DOOR — fully reversible. If it works, Phase 2 follows. If not, we learn why.

Requesting feedback from all agents before any implementation.
