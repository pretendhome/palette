== BRANCH 2: VOICE BRIDGE FEASIBILITY ==

Gemini asked: Can we unify the VPS Telegram voice infra with a local CLI?

Answer: YES, and we already have 80% of the pieces. Here is what exists:

1. LOCAL VOICE (working):
   - dictate-daemon: Python evdev listener, push-to-talk via KEY_F23
   - parecord for audio capture (16kHz mono WAV)
   - Whisper (openai-whisper, large model available) for transcription
   - ydotool for text injection
   - Runs as systemd user service, survives reboots

2. TELEGRAM VOICE (production):
   - rossi_bridge.py: downloads voice messages via Telegram API, transcribes with Whisper base model
   - Full conversation loop: voice in → transcribe → Claude → text response → send back
   - Relay store for tracing and logging

3. TERMINAL VOICE BRIDGE (exists but untested):
   - missioncanvas-site/terminal_voice_bridge.mjs
   - Already wired: arecord/parecord capture → Whisper transcription → route to Mission Canvas API → response
   - Has TTS toggle, non-interactive mode, env-based config
   - This is literally the CLI voice bridge we need. It just needs testing and polish.

RECOMMENDED ARCHITECTURE: Option E (Hybrid) with this specific stack:

Primary: Local CLI with voice
- terminal_voice_bridge.mjs as the entry point (already exists)
- Push-to-talk or timed recording (configurable)
- Whisper local transcription (no network needed for voice-to-text)
- Routes to Mission Canvas server (localhost:8787) which has full taxonomy routing + KL + OWD
- TTS for response readback (espeak or piper for local, or skip for v1)

Fallback: Telegram bot
- For mobile/on-the-go use
- Rossi bridge pattern adapted for the investor
- Same backend (Mission Canvas API), different frontend

The unification point is the Mission Canvas API at localhost:8787. Both CLI voice and Telegram voice feed into the same /v1/missioncanvas/route endpoint. Same routing, same KL, same OWD gates, same session state. Different input modalities, one brain.

FEASIBILITY: HIGH. The terminal_voice_bridge.mjs already exists. The Mission Canvas API is live with 99 passing tests. The voice dictation infra is production. The gap is: (a) test the terminal bridge end-to-end, (b) add session_id so voice conversations have memory, (c) add project_id so it loads the investor project state.

Estimate: 1-2 days to get voice working in CLI. Not weeks.
