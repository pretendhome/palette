10 Floors — The Version That Includes What Breaks
MCP is a universal standard that lets any AI tool call into your enterprise data through one integration instead of N×M custom connectors. What makes it enterprise-ready isn't the protocol — it's the permission model, the token scoping, and the admin controls around it. The five things that go wrong in real deployments are all at that boundary: the tool never gets called because the description is too vague for the LLM to find it, the response overwhelms the context window because nobody limited the result size, an admin token in production means everyone sees everything, the integration works in one host but breaks in another, or an API update silently returns wrong answers because the server parses the old format. None of these are protocol bugs. They're all operational — and they're all preventable if the SA has seen them before.

The 5 Things That Go Wrong

These are the real problems you will encounter in customer deployments. Memorize these.

1. The Tool Never Gets Called
No error. No log entry. The LLM just doesn't use the tool. Customer says "it's not working" and there's nothing to look at.
Root cause: The tool description is too vague. The LLM reads the description to decide whether to call the tool. "Searches stuff" won't trigger. "Search the enterprise knowledge base for documents, policies, and answers. Results respect user permissions." will.

Also check: Is the host configured to use the server? Is the server process running? Is the tool listed in tools/list?
2. The Response Blows Up
Enterprise search returns 10 full documents. The payload is huge. LLM context window overflows, truncates, or hallucinates. Customer sees a garbled or wrong answer.
Root cause: No result size control. The handler returns raw, unformatted API output or too many results.

Fix: Limit pageSize (5 is a good default). Truncate snippets (200 chars). Return structured summaries, not full documents. Format for readability, not completeness.
3. Everyone Can See Everything
User sees documents they shouldn't have access to. This is a security incident, not a bug.
Root cause: Admin token used instead of user-scoped token. Admin tokens bypass permission enforcement. During development, teams use admin tokens because they're easy. Nobody switches for production.

Fix: Production MCP servers MUST use user-scoped tokens via OAuth 2.1. the platform's managed MCP server handles this automatically. Local servers require explicit token scoping.

If this happens: Escalate immediately. Include token type, user email, document IDs visible.
4. It Works in Claude But Not in Cursor
Customer says "We tested in Claude Desktop and it works fine. But our team uses Cursor and it doesn't work there."
Root cause: MCP host implementations differ. Transport framing, capability negotiation, and notification handling vary across hosts. The spec is still evolving (current version: 2024-11-05).

Fix: Always test against the customer's preferred host before the demo. the platform's managed remote MCP server handles cross-host compatibility. Local servers may need per-host adjustments.
5. "It Worked Yesterday"
Integration was working. Product update changes an API response format. MCP server parses the old format. Results are silently wrong or empty.
Root cause: No version pinning or change detection between the MCP server and the API it wraps.

Fix: the platform's managed MCP server stays in sync with API changes automatically. For local/custom servers: pin to API version, monitor for changes, test after updates. Enablement content tied to release cycle flags this within 48 hours of a product change.

Diagnostic Decision Tree

When a customer reports an MCP issue, diagnose top-down.

Customer reports MCP issue
  │
  ├── Tool doesn't appear in host?
  │   └── Check: config path, JSON syntax, server process running → restart host
  │
  ├── Tool appears but never called?
  │   └── Check: tool description clear to the LLM? Prompt specific enough?
  │       ("Search the platform for..." not "find something about...")
  │
  ├── "Server disconnected" error?
  │   └── Check: Node 18+? Any console.log() in server? (breaks JSON-RPC protocol)
  │
  ├── Empty results or "Auth failed"?
  │   └── Check: API token set? Expired? Has MCP + SEARCH scopes?
  │
  └── User sees restricted documents?
      └── ESCALATE: admin token vs user-scoped? source-system ACLs synced?
          → Engineering with token type, user email, doc IDs
Never use console.log() in an MCP server. It writes to stdout, which is the JSON-RPC transport channel. One stray log statement breaks the entire protocol stream. Use process.stderr.write() for debugging. This is the #1 gotcha for every developer touching MCP for the first time.

The Security Conversation

This is the section you will use every day. Memorize these four answers.

"How do we ensure users only see documents they're authorized to access?"
the platform enforces permissions at the API layer. In production, each user authenticates via OAuth 2.1 and gets a user-scoped token. They only see documents they can access in the source systems. If you can't see it in SharePoint, you can't see it through MCP. The permission model is inherited, not configured separately.
"Who controls which MCP servers are available?"
Admin-managed configuration for enterprise AI tools — Claude Teams, Copilot Studio, ChatGPT Enterprise. End-user install for developer tools like Cursor and VS Code. Your IT team decides what's available, not individual users.
"How do we audit what the AI accessed?"
Every the platform API call is logged — including MCP-originated queries. The audit trail shows who asked, what was retrieved, which permissions applied, and when. This includes agent-initiated requests, not just human queries.
"Is MCP itself secure, or is it just a pipe?"
MCP is the transport. Security lives in two places: the platform's permission model (what the user can see) and admin configuration (which servers are available). The MCP server is where you enforce token scoping, input validation, and rate limiting. the platform's managed server handles all of this. Custom servers need to implement it.

Customer Talk Tracks

"Why MCP instead of a custom RAG integration?"
RAG requires you to build and maintain your own retrieval pipeline — chunking, embedding, vector store, reranking, permission enforcement, freshness tracking. MCP connects to the platform's existing knowledge graph, which already indexes your 100+ data sources with permissions enforced. You get RAG-quality answers without building RAG infrastructure.
"What's the business case?"
Teams get enterprise answers inside the tools they already use — Cursor, Claude, Copilot — without context switching. Faster onboarding, fewer "where do I find X?" tickets, higher adoption because the platform is embedded in daily workflows. And every MCP integration is a new surface for expansion: search today, document retrieval next, agent actions after that.
"Will MCP even exist in 2 years?"
Anthropic, OpenAI, Google, Microsoft, and the platform all support it today. If the standard evolves, the concepts transfer: tool definitions, schema-driven invocation, permission-aware context. The lab teaches the pattern, not just the protocol. And pragmatically — customers are asking about it now. We need to answer now.

Never start with architecture. Start with their problem, then show the solution working. Save the architecture diagram for when they ask "how does it work?" — not before.

Pre-Demo Checklist

  • Know which AI tools the customer's team uses (Cursor, Claude, Copilot, ChatGPT)
  • Know which data sources they search today (Confluence, Slack, Drive, SharePoint, Jira)
  • Know if they are evaluating or building RAG — the "build vs buy" conversation
  • MCP server running with a valid the platform token and tested query ready
  • Test the demo in the customer's preferred host, not just your preferred host
  • Backup: screenshots of a successful end-to-end flow in case of technical issues
  • Security answers memorized: permissions, admin control, audit trail, OAuth

Do Not Say

"Protocol" to non-technical buyers — say "standard" or "integration pattern" · "It's like an API" — it IS an API, but that undersells the universality · "Any host works" — without testing against that specific host first · "It's completely secure" — without specifying token scoping and permission model · "Just plug it in" — without discussing admin configuration and governance · console.log in any MCP server code, ever
