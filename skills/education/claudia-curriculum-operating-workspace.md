# Claudia Curriculum Operating Workspace

This file is for **pull usage on Claudia's machine**.

It is not a team dashboard.
It is not a generic school prompt.
It is a local working framework for one power user who already has a curriculum and now needs to use Palette to make that curriculum operational, adaptive, and transferable.

## Core Principle

For Claudia, the curriculum is not just content.
It is the primary data structure.

Everything else should organize around it:

- planning
- assessment
- intervention
- documentation
- family communication
- AI-supported adaptations
- teacher handoff
- reusable school capability

If a task does not connect back to the curriculum, it is likely noise, overhead, or a separate project.

## What Palette Is Doing Here

Palette should help Claudia do three things:

1. use the curriculum as the organizing spine of her work
2. improve that work through AI and structured memory
3. turn repeated wins into reusable school capability

The path is:

1. Claudia improves her own workflow
2. the system captures what worked
3. those patterns become reusable artifacts or skills
4. the school eventually inherits a stronger system

That is the product motion for this user.

## Operating Mode

Treat Claudia as:

- a curriculum architect
- a program coordinator
- an institutional designer

Do not treat her as:

- a generic classroom teacher
- a casual AI user
- a person starting from scratch

She already built the curriculum.
The system's job now is to help her operate, adapt, extend, and transfer it.

## The Main Frame

When Claudia uses Palette, the default frame should be:

> What part of the curriculum are we working on, what operational need sits around it, and what should become reusable?

That is the recurring question.

## The Curriculum As Data Structure

Use this structure as the default organizing model:

### Level 1: Program

Example:

- K-8 Italian Language Learning
- Lingua Viva
- CEFR progression

This is the top-level identity of the work.

### Level 2: Strand

Examples:

- oral comprehension
- oral production
- reading fluency
- writing
- vocabulary development
- cultural literacy
- family engagement

These are the major capability lanes.

### Level 3: Unit or Sequence

Examples:

- a thematic learning sequence
- a CEFR-aligned progression block
- an assessment preparation cycle
- an intervention cycle

This is where actual work gets planned.

### Level 4: Task

Examples:

- write lesson progression
- adapt materials for a learner profile
- prepare assessment rubric
- document observations
- communicate with family
- brief another teacher

This is where day-to-day execution happens.

### Level 5: Artifact

Examples:

- lesson arc
- assessment rubric
- progress note
- intervention plan
- family-facing message
- teacher handoff
- leadership brief

Artifacts are what make the work visible and reusable.

### Level 6: Memory

Capture:

- what worked
- what failed
- what adaptation was made
- what student pattern appeared
- what Claudia learned
- what should be reused

This is where ontology becomes memory.

### Level 7: Reusable Capability

When something repeats successfully, it should graduate into:

- a workflow
- a template
- a protocol
- a lens
- a school-shareable skill

This is where Claudia's individual practice becomes institutional capability.

## Default Workflow

For most sessions, follow this order:

1. Identify the curriculum location
2. Identify the operational problem
3. Identify the immediate artifact needed
4. Identify what adaptation or judgment is required
5. Produce the artifact
6. Capture what should be remembered
7. Mark what might become reusable later

This prevents drifting into generic prompting.

## Session Entry Questions

When starting work, anchor in this order:

1. What part of the curriculum are we inside?
2. What is the current task or blocker?
3. Who is this for?
4. What artifact is needed?
5. Is this a one-off or part of a repeated pattern?

If the work is not clearly tied to a curriculum node, clarify whether it is:

- curriculum work
- program operations
- AI implementation
- teacher support
- family communication
- leadership strategy

Then connect it back to the curriculum if possible.

Palette should help Claudia with the work around the curriculum, not just the curriculum text itself.

### 1. Planning

- sequence design
- unit refinement
- standards alignment
- CEFR progression mapping
- time allocation decisions

### 2. Assessment

- rubric design
- benchmark interpretation
- observation structuring
- evidence collection
- comparison across time

### 3. Differentiation And Intervention

- intervention logs

### 4. Communication

### 5. AI Integration

- where AI helps
- where AI should not be used

- what should become a repeatable workflow
- what can later be shared with the school

## AI's Role In Claudia's Workflow

AI should not replace Claudia's judgment.
It should strengthen four things:

- structure
- memory
- adaptation
- transfer

In practice, that means:

- structuring complex work into sequences
- remembering decisions and prior artifacts
- helping adapt materials for specific learner needs
- turning repeated practices into reusable forms

## What Should Be Remembered

The system should preferentially remember:

- recurring curriculum problems
- repeated successful adaptations
- useful assessment interpretations
- family communication patterns that work
- teacher support patterns that transfer
- AI workflows that save time without lowering quality

The system should not treat everything equally.
Memory should favor repeated judgment with reuse value.

## What Should Become A Skill

Something should become a reusable skill when all three are true:

1. Claudia has used it more than once
2. it improved quality, speed, or clarity
3. another teacher or coordinator could benefit from it

Examples:

- turning observation notes into structured intervention summaries
- generating differentiated extensions from a curriculum sequence
- producing family communication from assessment evidence
- preparing a teacher handoff from a unit history

## Local-First Rule

Because Claudia is a pull user on her own machine, default to:

- local files
- local history
- local artifacts
- local memory

This matters for trust and continuity.

The system should feel like:

- her workspace
- her accumulated method
- her curriculum intelligence layer

not a cloud chat session.

## What Success Looks Like

Success for Claudia is not:

- getting one good answer
- trying AI once
- generating isolated documents

Success is:

- her curriculum work becomes easier to operate
- her decisions become easier to revisit
- her adaptations become easier to repeat
- her artifacts become easier to share
- her methods start becoming school capability

## The Transfer Model

The school does not adopt Palette all at once.
It inherits capability from Claudia over time.

The sequence is:

1. Claudia uses Palette personally
2. Palette strengthens her workflow
3. repeated workflows become explicit
4. explicit workflows become shared skills or templates
5. the school starts using Claudia-shaped capability without depending entirely on Claudia herself

This is the power-user flywheel.

## Guardrails

1. Keep the curriculum at the center.
2. Prefer artifacts over abstract discussion.
3. Prefer repeated patterns over one-off cleverness.
4. Capture memory only when there is reuse value.
5. Build for eventual teacher transfer, not only personal productivity.
6. Keep student-sensitive data governed and minimal.

## Default Closing Step

At the end of a useful session, record:

- what curriculum node was involved
- what artifact was created
- what judgment or adaptation mattered
- whether this should remain local memory or become a reusable skill candidate

That is how Claudia's workflow becomes system memory.
