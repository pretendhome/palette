## Prologue: What the Code Reveals

Before answering the five questions, here is what the codebase analysis reveals about the current state of the loop:

### What Already Works (and is production-proven)

The 5-step pipeline in `palette_query.py` is fully implemented:

```
RESOLVE → RETRIEVE → ROUTE → RESPOND → EXTRACT
```

- **`step_resolve()`**: calls `palette_retrieve.retrieve()` — hybrid FTS5 + keyword + vector, returns `riu_id`, `confidence`, `classification`
- **`step_retrieve()`**: extracts `knowledge[]` from resolve result — no separate call, piggybacks on resolve
- **`step_route()`**: maps `classification → agent` via `CLASSIFICATION_TO_AGENT` dict, calls `bus_send()`, runs `query_before_act` CQ check
- **`step_respond()`**: formats grounded output — knows about `gateway` result or falls back to `knowledge[]`
- **`step_extract()`**: generates a `gap_signal` if confidence < 30, or `retrieval_success` if ≥ 70; broadcasts both to bus

The gateway (`bdb/gateway/__init__.py`) has a real confidence gate: `needs_external()` returns `True` only when `confidence < self.threshold` (default 40). PII is stripped by `sanitizer.py` before anything exits. The broker (`broker/index.mjs`) validates every envelope against `validate.mjs` before inserting — `message_type`, `risk_level`, `trust_tier`, UUID format, `requires_ack` coercion for high-risk messages.

### What's Missing (the gap between "works" and "obligatory")

1. **The routing step is a suggestion, not a gate.** `step_route()` calls `bus_send()` — which is `best-effort` and non-blocking. Comment in `palette_query.py`: `"bus is nice-to-have for logging/governance, not blocking. The query pipeline works without the bus (retrieval is local)"`. The classification happens, but nothing enforces that downstream work actually receives it.

2. **`auto_enrich.py` doesn't exist at `scripts/palette_intelligence_system/auto_enrich.py`.** That path returned 404. The self-improvement seeds that do exist are: `coordination.py` (4-step coordinator), `query_engine.py` (gap detection), and `health_check.py` (84 integrity checks). These are each individually capable, but disconnected from each other and from the main query loop.

3. **The EXTRACT step produces gap signals, but they go to the bus as broadcast messages.** Nothing reads and acts on them. The bus has FTS5 search and per-agent memory — but no agent is subscribed to read gap signals and propose taxonomy updates.

4. **No RIU-classification field in the bus envelope schema.** `validate.mjs` requires: `protocol_version`, `message_id`, `thread_id`, `from_agent`, `to_agent`, `message_type`, `intent`, `risk_level`, `requires_ack`, `created_at`, `payload`. The `payload` is an open object. There is no required `riu_id` or `confidence` field — any agent can send a message that bypasses taxonomy classification entirely.

5. **Voice Hub (server.mjs) calls `palette_retrieve.py` as a subprocess.** This is the correct pattern. The gap is that responses generated by the Voice Hub aren't fed back through EXTRACT/STORE — the compounding loop is broken for voice queries.

These five gaps define exactly what "making the loop obligatory" requires. Each is a wiring change, not a rewrite.

## Q1 — Obligatory Routing: Making It an OS, Not a Suggestion

### The OS Kernel Analogy — and Why It's the Right Frame

A kernel makes system calls obligatory through a **privilege boundary**: user-space code physically cannot access hardware without crossing the boundary, because the hardware enforces it. The boundary isn't a convention — it's architecture.

DNS makes name resolution obligatory through **infrastructure dependency**: you can't reach an IP without resolving a name, because the network stack doesn't accept raw hostnames at the routing layer.

OAuth makes authorization obligatory through **token requirement**: a resource server returns 401 unless a valid bearer token is present, and the token can only be obtained through the auth flow.

The pattern across all three: **the protected resource cannot be accessed without going through the mandatory layer.** The layer is not a wrapper — it's a gate in the data path itself.

For Palette, the equivalent is: **no agent produces a response to a query unless that query has been classified by the taxonomy first.** The classification result (RIU ID, confidence, classification tier) is not optional context — it is a required field in every work envelope.

### The MCP Envelope Pattern — Your Bus Already Has It

`validate.mjs` already implements mandatory field validation. Every message that fails validation gets a 400 response with error details — the message is never stored, the agent never receives it. This is the enforcement mechanism.

The current required fields include `message_type` (one of 7 values), `risk_level` (one of 5 values), and `requires_ack`. What's missing is `riu_id` as a required field for any `execution_request` or `advisory` type message.

**The fix: add RIU classification as a required payload field for execution-class messages.**

In `validate.mjs`:

```javascript
// Add to validateEnvelope:
const EXECUTION_TYPES = ['execution_request', 'advisory', 'proposal'];

if (EXECUTION_TYPES.includes(env.message_type)) {
  const payload = env.payload || {};
  if (!payload.riu_id) {
    errors.push('execution_request payload must include payload.riu_id (taxonomy classification required)');
  }
  if (payload.confidence == null) {
    errors.push('execution_request payload must include payload.confidence');
  }
  if (payload.confidence != null && payload.confidence < 0) {
    errors.push('payload.confidence must be >= 0');
  }
}
```

This change means: **any agent that tries to request execution without first classifying through the taxonomy will receive a 400 error from the bus.** The bus becomes the enforcement point. No agent can skip Step 1 because the message that would trigger Step 4 will be rejected without it.

This is the kernel pattern: the resource (agent execution) cannot be accessed without going through the gate (taxonomy classification). The gate is in the infrastructure.

### Claude Code Hooks — The PreToolUse Pattern

[Claude Code's `PreToolUse` hook](https://code.claude.com/docs/en/hooks) fires before every tool call executes and can block it deterministically. The hook receives the tool name and input parameters on stdin as JSON, and returns a `permissionDecision: "deny"` to block.

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "Query not classified. Route through palette_retrieve first.",
    "additionalContext": "RIU classification is required before tool execution in Palette."
  }
}
```

For Palette operating inside Claude Code (as a development environment), this means:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "mcp__palette__.*",
        "hooks": [
          {
            "type": "http",
            "url": "http://localhost:7899/hooks/pre-tool-use",
            "timeout": 5
          }
        ]
      }
    ]
  }
}
```

The broker can expose a `/hooks/pre-tool-use` endpoint that checks whether the incoming tool call payload includes a valid `riu_id`. If not: deny. If yes: allow and inject the classification context via `additionalContext`.

This makes the Claude Code layer enforce taxonomy-first even for ad-hoc tool calls that bypass the CLI pipeline. The hook fires on every MCP tool call from any pattern matching `mcp__palette__.*`.

### The Tool Attention Pattern (arxiv, April 2026)

The [Tool Attention paper (arxiv 2604.21816)](https://arxiv.org/html/2604.21816v1) introduces a middleware-layer mechanism sitting in the `before_model` hook of LangGraph/LangChain/Semantic Kernel. It gates tool availability based on query-tool semantic overlap — tools that aren't relevant to the current query are excluded from the model's prompt entirely.

The key architectural insight: **the hallucination rejection gate** runs in `after_model` — if the model tries to call a tool that wasn't promoted this turn (the model saw a summary but not the full schema), the middleware rejects the call with a structured error. This is deterministic enforcement, not probabilistic hope.

Mapping to Palette: the taxonomy classification step is exactly the `before_model` hook — it determines which knowledge is relevant before the model sees the prompt. The broker's envelope validation is the `after_model` gate — if the execution_request lacks a `riu_id`, it's rejected with a structured error.

The paper's formulation applies directly:

> "Any false negative at the routing layer is caught deterministically downstream."

Palette already has the downstream gate (broker validation). The upstream routing (taxonomy classification) already runs. The missing piece is making the downstream gate reject messages that skipped the upstream routing — which the `validate.mjs` change above implements.

### OpenAI Agents SDK — Blocking Input Guardrails

The [OpenAI Agents SDK](https://openai.github.io/openai-agents-python/guardrails/) supports blocking input guardrails: `run_in_parallel=False` makes the guardrail complete before the agent starts. If `tripwire_triggered = True`, the agent never executes.

For Palette's multi-agent coordination (coordination.py), this means wrapping the `researcher` and `final` steps with a guardrail that checks for `resolved_rius` in the handoff packet:

```python
@input_guardrail(run_in_parallel=False)
async def require_riu_classification(ctx, agent, input):
    packet = input.get("handoff_packet", {})
    if not packet.get("resolved_rius"):
        return GuardrailFunctionOutput(
            output_info="No RIU classification in handoff packet",
            tripwire_triggered=True
        )
    return GuardrailFunctionOutput(output_info="Classification present", tripwire_triggered=False)
```

The `coordination.py` pipeline already has a 4-step flow: `resolver → traversal → researcher → final`. The resolver step already populates `resolved_rius`. Adding this guardrail to the `researcher` step makes it physically impossible to reach the researcher without going through the resolver.

### LangGraph — The Guard Node Pattern

In [LangGraph](https://docs.langchain.com/oss/python/langgraph/thinking-in-langgraph), the guard node is the entry point of the graph. Every query enters through the guard node first. The guard node adds a conditional edge: if classification succeeds, route forward; if not, route to an error handler.

```python
graph.set_entry_point("taxonomy_classify")
graph.add_conditional_edges(
    "taxonomy_classify",
    lambda state: "proceed" if state["confidence"] > 0 else "no_classification",
    {
        "proceed": "knowledge_retrieve",
        "no_classification": "return_error"
    }
)
```

Palette doesn't use LangGraph today, but the pattern is directly implementable in the existing Python pipeline. `palette_query.py::run_query()` calls steps sequentially. Making it obligatory means: if `step_resolve()` returns `confidence == 0` AND `riu_id is None`, raise an exception before proceeding to `step_route()`.

Currently `step_route()` handles `None` RIU gracefully — it defaults to `claude.analysis`. That's the exact behavior that needs to change. A query with no classification should not route at all; it should return a "classification failed" result and log a gap signal.

### The Synthesis — What "Obligatory" Means in Practice

The correct architectural frame is not "add a check somewhere." It's: **the classification result is the key that unlocks the rest of the pipeline.** Every downstream step consumes the key. If the key is absent, the step fails with a structured error, not silently.

Three layers enforce this:

1. **Envelope schema** (broker/validate.mjs): no `execution_request` without `riu_id` + `confidence`
2. **Pipeline guard** (palette_query.py): no routing if `confidence == 0` and `riu_id is None`
3. **HTTP hook** (Claude Code PreToolUse): no MCP tool call without classification in payload

All three are wiring changes in existing files. Total estimated effort: 1 day.

## Q2 — Gap-Filling: Targeted Perplexity Queries from Taxonomy State

### What the Gateway Already Does

`bdb/gateway/__init__.py::needs_external()` checks: `confidence < self.threshold` (default 40). When true, `query_perplexity()` fires with the raw sanitized query.

The current system prompt sent to Perplexity is:
```
"You answer public legal research questions only. Do not infer client-specific advice. Cite public sources when possible."
```

This is generic. The query is the sanitized user input. There is **no taxonomy context** in the Perplexity call. The system doesn't tell Perplexity what it already knows, what it's missing, or what the RIU classification was.

This is a significant missed opportunity. A targeted search query using what the taxonomy already knows will return better results than an open-ended search using the raw user question.

### The Skill-RAG Pattern (arxiv 2604.15771, April 2026)

[Skill-RAG](https://arxiv.org/html/2604.15771v1) identifies a "query–evidence alignment gap": the query is poorly formulated relative to the evidence space. When retrieval confidence is low, it's not always because the answer doesn't exist — it's often because the query doesn't match how the evidence is indexed.

Skill-RAG's solution: when retrieval confidence falls below threshold, a skill router diagnoses WHY and selects one of four skills:

| Skill | When to Use | Effect |
|---|---|---|
| Query rewriting | Surface form diverges from evidence indexing | Reformulate to match corpus vocabulary |
| Question decomposition | Entangled premises | Break into sub-questions |
| Evidence focusing | Query too broad | Extract specific missing slot and query for it |
| Exit | Truly irreducible gap | Return what's known, flag the gap |

Palette's taxonomy already does implicit classification — the RIU ID tells you the problem type, and the confidence score tells you how well the retrieved knowledge matches. What's missing is the diagnostic step that identifies WHICH of these four failure modes applies.

The implementation maps directly: when `confidence < 40`:

```python
def _formulate_targeted_query(query: str, retrieval_result: dict) -> str:
    riu_id = retrieval_result.get("riu_id")
    riu_name = retrieval_result.get("riu_name", "")
    confidence = retrieval_result.get("confidence", 0)
    knowledge = retrieval_result.get("knowledge", [])
    
    # Build what-we-know context
    known_entries = [k["question"] for k in knowledge[:3] if k.get("question")]
    known_summary = "; ".join(known_entries) if known_entries else "no relevant knowledge found"
    
    # Build gap description
    if confidence < 15:
        gap_type = "completely absent from knowledge library"
    elif confidence < 30:
        gap_type = "partially covered — missing key precedents or recent developments"
    else:
        gap_type = "covered but confidence is marginal — may need current sources"
    
    # Formulate targeted query
    targeted = (
        f"Context: This question relates to {riu_name} ({riu_id}). "
        f"Known: {known_summary}. "
        f"Knowledge gap: {gap_type}. "
        f"Query: {query}"
    )
    return targeted
```

### The Perplexity System Prompt as Context Carrier

The `query_perplexity()` method currently hardcodes a generic system prompt. The fix: build a taxonomy-aware system prompt that:

1. Declares the RIU classification (tells Perplexity what kind of question this is)
2. States what the local knowledge library already has (prevents redundant answers)
3. States what the gap is (focuses the search on what's actually missing)
4. Requests structured output that maps to KL entry format

```python
def _build_targeted_system_prompt(retrieval_result: dict) -> str:
    riu_id = retrieval_result.get("riu_id", "unknown")
    riu_name = retrieval_result.get("riu_name", "")
    confidence = retrieval_result.get("confidence", 0)
    knowledge = retrieval_result.get("knowledge", [])
    
    known_questions = [k["question"] for k in knowledge[:3] if k.get("question")]
    
    system_prompt = (
        f"You are answering a {riu_name} ({riu_id}) query for a governed AI system.\n"
        f"Local confidence: {confidence:.0f}%. "
        f"{'Answer is partially known but needs supplementation.' if confidence > 20 else 'No local answer found.'}\n"
    )
    
    if known_questions:
        system_prompt += f"Already known: {'; '.join(known_questions[:2])}. Do not repeat this.\n"
    
    system_prompt += (
        "Focus on: public sources, authoritative citations, recent developments (2024-2026).\n"
        "Return: direct answer with citations. No client-specific advice."
    )
    return system_prompt
```

**In `gateway_query()`**, replace the hardcoded system prompt in `query_perplexity()` with this targeted prompt:

```python
external = self.query_perplexity(
    sanitized_query, 
    system_prompt=self._build_targeted_system_prompt(retrieval_result)
)
```

### Perplexity Sonar Pro API — The Right Parameters

The Sonar Pro API (`sonar-pro` model) supports:
- **200K context window** — enough to include full taxonomy classification + 5 KL entries as system context
- **Structured outputs** via `response_format.type = "json_schema"` — force Perplexity to return a JSON object that maps directly to a KL entry candidate
- **Real-time web search** — the answer is grounded in current sources, not training data

The structured output pattern turns the search result directly into a KL entry proposal:

```python
KL_ENTRY_SCHEMA = {
    "type": "object",
    "properties": {
        "question": {"type": "string"},
        "answer": {"type": "string"},
        "evidence_tier": {"type": "integer", "enum": [1, 2, 3]},
        "sources": {"type": "array", "items": {"type": "string"}},
        "tags": {"type": "array", "items": {"type": "string"}},
        "related_rius": {"type": "array", "items": {"type": "string"}}
    },
    "required": ["question", "answer", "evidence_tier", "sources"]
}

# In query_perplexity():
payload["response_format"] = {
    "type": "json_schema",
    "json_schema": {
        "name": "kl_entry_proposal",
        "schema": KL_ENTRY_SCHEMA
    }
}
```

When Perplexity returns a structured JSON object that already maps to the KL schema, the gap between "search result" and "KL entry proposal" collapses. The auto-enrich pipeline becomes trivial: validate, add `proposed_at` timestamp, add `riu_id`, write to a proposals file for human review.

### The ConfRAG Pattern — Confidence-Guided Query Expansion

[ConfRAG (OpenReview 2026)](https://openreview.net/forum?id=1OLTJL1wHG) proposes confidence-guided retrieval augmentation: when initial retrieval confidence falls below threshold, rather than just searching for the original query, the system uses the retrieved fragments as context to generate a better-targeted search query.

The Palette equivalent uses the partial KL matches as "fragments" to construct a more targeted query:

```python
def _expand_from_fragments(query: str, knowledge: list[dict], riu_id: str) -> str:
    """Use partial KL matches to construct a more targeted search."""
    fragment_context = " ".join([
        k.get("answer_excerpt", "")[:100] 
        for k in knowledge[:2] 
        if k.get("answer_excerpt")
    ])
    
    if not fragment_context:
        return query
    
    # The query is not "what is X" but "what did X decide about Y, given that we already know Z"
    return (
        f"Given that: {fragment_context[:200]}. "
        f"What is missing from this picture regarding: {query}? "
        f"Provide authoritative sources (case law, regulations, official guidance) "
        f"specific to {riu_id}."
    )
```

This is the "evidence focusing" skill from Skill-RAG — extract what's known from fragments, formulate a query that fills the specific gap, not the whole question.

### Implementation Summary

In `bdb/gateway/__init__.py`, three changes:

1. Add `_formulate_targeted_query()` method
2. Add `_build_targeted_system_prompt()` method
3. Add `KL_ENTRY_SCHEMA` and pass to `query_perplexity()` when `retrieval_result` has a `riu_id`

In `query_perplexity()`, update `payload["messages"][0]["content"]` to use the targeted system prompt. Pass `response_format` when schema is provided.

Estimated effort: 4-6 hours.

## Q3 — Multi-Agent Convergence with the Current Relay Model

### What the Bus Already Provides

`broker/index.mjs` has every primitive needed for convergence:

- **Thread tracking**: every message has a `thread_id` UUID — the entire conversation across agents is queryable via `POST /thread`
- **Per-agent memory**: `POST /memory` with bounded storage (2200 chars/agent) — agents can persist their sub-conclusions across turns
- **Skill sharing**: agents can publish `WORKING` or `PRODUCTION` skills to a shared pool, other agents adopt them
- **FTS5 message search**: `POST /search` queries all messages across all agents by content
- **Skills maturity auto-promotion**: UNVALIDATED → WORKING (3 impressions, <50% failure) → PRODUCTION (10 impressions, <20% failure)

The broker already supports convergence state. The problem is the relay model (Codex designs → Kiro builds → Claude finishes) is sequential — there's no fan-out or merge.

### The Hub-and-Spoke + Parallel Fan-Out Pattern

[OpenAI's multi-agent portfolio example](https://developers.openai.com/cookbook/examples/agents_sdk/multi-agent-portfolio-collaboration/multi_agent_portfolio_collaboration) demonstrates the hub-and-spoke model: one orchestrator agent (Portfolio Manager) dispatches to parallel specialists (Fundamental, Macro, Quantitative) and merges their outputs. The specialists run concurrently via `parallel_tool_calls=True`.

This maps directly to Palette's relay:

| Current (Sequential) | Target (Parallel with Convergence) |
|---|---|
| Codex designs → Kiro builds → Claude finishes | Orchestrator classifies via taxonomy → fans out to Codex + Kiro + Claude in parallel → convergence agent merges |
| Each agent starts blind | Each agent receives classified problem + KL entries + prior agent outputs as context |
| No deduplication | Convergence agent detects overlapping outputs |
| No contradiction detection | Convergence agent flags divergence above threshold |

The key change: **the classified problem (RIU ID + confidence + KL entries) is the shared context that every parallel agent receives.** Without taxonomy classification, parallel agents start from different assumptions and produce incoherent outputs. With it, they share a semantic frame.

### The HandoffPacket Pattern Already in coordination.py

`coordination.py` already implements `HandoffPacket v2` with `resolved_rius` as a first-class field. The current 4-step flow is sequential: `resolver → traversal → researcher → final`.

Making it parallel requires branching after `traversal`:

```python
# In coordination.py, after traversal step:
if packet.get("resolved_rius"):
    # Fan out to parallel agents
    subtasks = _decompose_into_subtasks(packet)  # uses RIU classification
    results = await asyncio.gather(*[
        _run_agent_subtask(agent, subtask, packet)
        for agent, subtask in subtasks
    ])
    # Convergence: merge results, detect contradictions
    packet["outputs"]["parallel_results"] = results
    packet["outputs"]["convergence"] = _convergence_merge(results)
```

The `_decompose_into_subtasks()` function uses the RIU classification to assign work:

```python
RIU_TO_AGENT_ASSIGNMENT = {
    "architecture": ["codex.implementation", "claude.analysis"],
    "research": ["perplexity.computer", "claude.analysis"],
    "build": ["kiro.design", "codex.implementation"],
    "review": ["claude.analysis", "mistral.vibe"],
    "legal": ["claude.analysis", "perplexity.computer"],
}
```

### Contradiction Detection and Convergence

[LangGraph's parallel node pattern](https://www.linkedin.com/posts/giuseppemurro_parallel-nodes-in-langgraph-managing-concurrent-activity-7384913596180336640-OFJX) requires a synchronization barrier (deferred execution) to prevent race conditions. In Palette, the broker's thread model provides this naturally — all parallel messages share a `thread_id`, and the convergence step reads the full thread via `POST /thread`.

The convergence function checks for:

1. **Factual contradiction**: two agents make incompatible claims about the same question
2. **Coverage overlap**: two agents produced the same output (deduplication)
3. **Coverage gap**: a sub-problem was assigned but no agent addressed it
4. **Confidence divergence**: agents have very different confidence in their outputs

For Palette's context (professional decision support), a simple heuristic works: if agents disagree on a ONE-WAY DOOR item (as flagged by `palette-core.md`), escalate to human review via the broker's `waiting_human` state. If they agree, merge and proceed.

The broker already has `POST /approve` and `POST /reject` for human review of `waiting_human` messages. This is the existing ONE-WAY DOOR gate — it just needs to be triggered by convergence failures.

### Voice Hub Integration (server.mjs)

The Voice Hub calls `palette_retrieve.py` as a subprocess today. The problem is the subprocess call returns a result, but the Voice Hub doesn't feed that result back through the EXTRACT step. The compounding loop is broken for voice.

The fix: after Voice Hub receives a retrieval result, it must POST to the broker with the result attached to the query's thread. This makes voice queries first-class citizens of the session log.

```javascript
// In server.mjs, after subprocess result:
await fetch('http://127.0.0.1:7899/send', {
  method: 'POST',
  headers: {'Content-Type': 'application/json'},
  body: JSON.stringify({
    protocol_version: '1.0.0',
    message_id: crypto.randomUUID(),
    thread_id: sessionThreadId,
    from_agent: 'voice.hub',
    to_agent: 'group',
    message_type: 'informational',
    intent: `voice query completed: ${result.riu_id} (${result.confidence.toFixed(0)}%)`,
    risk_level: 'none',
    requires_ack: false,
    payload: {
      query: query,
      riu_id: result.riu_id,
      confidence: result.confidence,
      extraction: result.confidence < 30 ? 'gap_signal' : 'retrieval_success'
    },
    created_at: new Date().toISOString(),
    ttl_seconds: 3600
  })
});
```

## Q4 — Self-Improving Knowledge: Compound, Not Just Accumulate

### The Distinction That Matters

The [A-Mem agentic memory system (OpenReview 2025)](https://openreview.net/forum?id=FiM0M8gcct) makes the distinction precisely:

> "If memory remains unchanged, then the agents don't genuinely enhance their capabilities; they merely become faster at retrieving information."

Accumulation = adding entries. Compounding = entries that reference each other, update each other, and make future retrievals more accurate than past ones.

Palette's Knowledge Library at v1.4 has 183 entries and 565 citations. The entries exist. The question is whether they compound — whether using entry A to answer a question makes entry B more accurate next time a related question arrives.

The answer today is: **no**. Entries are static YAML. Health checks verify count and schema. Nothing updates entry quality based on use.

### The Self-Improvement Architecture — What Connects to What

Here is the current state of the self-improvement seeds in the repo, and where they need to connect:

**`health_check.py`** (84 integrity checks):
- Already checks: taxonomy count vs MANIFEST, KL count vs MANIFEST, agent schema validity, PIS integrity, routing coverage
- Does NOT check: entry quality, citation freshness, retrieval success rates, gap signal frequency

**`coordination.py`** (4-step pipeline):
- Has `_new_packet()` with a `gaps: []` field
- The traversal step outputs gaps as part of its result
- These gaps are stored in the task JSON but never aggregated

**`query_engine.py`** (gap detection):
- Has a `gaps` command that identifies RIUs without coverage
- Does NOT read from the session log or broker messages to identify runtime gaps

**`palette_query.py::step_extract()`** (gap signal generation):
- Generates `gap_signal` when `confidence < 30`
- Broadcasts to bus via `bus_send(to_agent="group", intent="Content gap detected...")`
- Nobody reads these messages

The self-improvement loop requires connecting these four components:

```
session_log.ndjson → gap_signal aggregator → gap_cluster detector 
→ KL entry proposer → human review gate → KL entry commit
```

### The Four Update Types and Their Governance Rules

The North Star Vision identifies five self-improvement signals. Here is each one mapped to existing code and the governance rule it requires:

**1. New KL entries from successful external searches**

*Trigger*: `step_extract()` logs `retrieval_success` with a gateway result
*What to do*: the gateway's Perplexity response (especially with structured output as per Q2) is already a KL entry candidate
*Governance rule*: **human review required** before commit (TWO-WAY DOOR action — reversible, but adds to the KL)
*Implementation*: write Perplexity structured response to `knowledge-library/proposals/YYYY-MM-DD-{riu_id}.yaml`; health_check.py flags unreviewed proposals as a warning

**2. New RIU nodes when taxonomy can't classify a problem**

*Trigger*: `step_extract()` logs `gap_signal` — confidence < 30, meaning no existing RIU matched well
*What to do*: aggregate gap signals by semantic similarity; if 3+ similar gaps cluster → propose new RIU
*Governance rule*: **human approval required** before adding (ONE-WAY DOOR — taxonomy structure is foundational)
*Implementation*: gap_aggregator.py reads `session_log.ndjson`, clusters gap signals by intent similarity (FTS5 search on broker `POST /search`), writes proposals to `taxonomy/proposals/`

**3. Evidence tier updates when better sources are found**

*Trigger*: Perplexity returns a Tier 1 source (court ruling, statute, official guidance) for a query that was answered by a Tier 2 or 3 KL entry
*What to do*: propose upgrading the existing entry's `evidence_tier` and adding the new source to its citations
*Governance rule*: **automated** if upgrading tier (Tier 3→2 or 2→1); **human review** if downgrading
*Implementation*: in `gateway_query()`, after merging external results, check if external source quality exceeds local entry quality; if so, write an upgrade proposal

**4. Agent maturity updates based on output quality**

*Trigger*: broker's `POST /skills` with `action: record_use` — already fires when a skill is used
*What to do*: auto-promote UNVALIDATED → WORKING → PRODUCTION based on success thresholds (already implemented in broker)
*Governance rule*: **fully automated** (already implemented: `MATURITY_THRESHOLDS` in broker index.mjs)
*Status*: **already works**. This is the one self-improvement loop that is already complete. UNVALIDATED skills promote to WORKING at 3 impressions with < 50% failure, WORKING to PRODUCTION at 10 impressions with < 20% failure.

**5. Retrieval quality improves when gap signals cluster**

*Trigger*: multiple gap signals with similar RIU and similar query content
*What to do*: signal that the FTS5 index may need re-weighting for that RIU cluster, or that new KL entries are needed
*Governance rule*: **human review** before index rebuild (an index rebuild could degrade retrieval quality for other queries)
*Implementation*: gap_aggregator.py counts gap signals per RIU over rolling 7-day window; if a single RIU has > 5 gap signals, emit a `taxonomy_gap_alert` to the bus; health_check.py flags it as a warning

### The A-Mem Compounding Pattern

[A-Mem](https://openreview.net/forum?id=FiM0M8gcct) implements compounding memory by:

1. When a new entry is added, generate structured attributes: contextual description, keywords, tags
2. Analyze historical entries for relevant connections
3. Establish links between the new entry and related historical entries
4. When a related entry is linked, update its contextual representation to include the new connection

This is the Zettelkasten method applied to agent memory. Palette's KL is already YAML with `related_rius` and `tags` fields. The links exist structurally. What's missing is the dynamic update: when a new KL entry is added, automatically scan existing entries for semantic similarity and add bidirectional `related_rius` links.

The FTS5 index in `kl_fts.db` can do this: after adding a new entry, run an FTS5 query on its content against existing entries, and add matches above a similarity threshold as related entries. This is a ~20-line Python function.

### The CDR Framework — Preventing Quality Degradation

The [CSA Cognitive Degradation Resilience (CDR) framework](https://cloudsecurityalliance.org/blog/2025/11/10/introducing-cognitive-degradation-resilience-cdr-a-framework-for-safeguarding-agentic-ai-systems-from-systemic-collapse) identifies "memory entrenchment" as the primary quality degradation risk: poisoned data writes into long-term memory and propagates into future recalls.

For Palette, the guard against this is the **evidence tier system**. A Tier 3 entry (community knowledge, no citations) cannot override a Tier 1 entry (court ruling, statute). The retrieval ranking in `hybrid_retrieve()` already applies tier boosts:

```python
tier_boost = {1: 1.2, 2: 1.0, 3: 0.9}.get(tier, 1.0)
```

The self-improvement governance rule extends this: **new entries proposed by automated processes start at Tier 3.** Human review can promote them. They can never be automatically promoted to Tier 1 — that tier requires a human to have verified the primary source.

This is the CDR "Memory Integrity Enforcement" control (BC-007): tier hierarchy as a quarantine boundary.

### The Missing `auto_enrich.py` — What to Build

The path `scripts/palette_intelligence_system/auto_enrich.py` returned 404. This file needs to be created. Based on the existing code patterns (coordination.py, query_engine.py, health_check.py), the minimal implementation:

```python
#!/usr/bin/env python3
"""auto_enrich.py — Gap-signal aggregator and KL entry proposer.

Reads session_log.ndjson and broker gap signals.
Clusters gap signals by RIU.
Proposes new KL entries from accumulated search results.
Writes proposals to knowledge-library/proposals/ for human review.
"""

import json
from pathlib import Path
from datetime import datetime, timezone

REPO_ROOT = Path(__file__).resolve().parents[2]
SESSION_LOG = REPO_ROOT / "peers" / "session_log.ndjson"
PROPOSALS_DIR = REPO_ROOT / "knowledge-library" / "proposals"

def read_gap_signals(days: int = 7) -> list[dict]:
    """Read gap signals from session log within rolling window."""
    signals = []
    if not SESSION_LOG.exists():
        return signals
    cutoff = datetime.now(timezone.utc).timestamp() - (days * 86400)
    with open(SESSION_LOG) as f:
        for line in f:
            try:
                entry = json.loads(line.strip())
                if entry.get("extraction") == "gap_signal":
                    # Check timestamp
                    ts = entry.get("timestamp", "")
                    if ts:
                        from datetime import datetime
                        dt = datetime.fromisoformat(ts.replace("Z", "+00:00"))
                        if dt.timestamp() > cutoff:
                            signals.append(entry)
            except (json.JSONDecodeError, KeyError, ValueError):
                pass
    return signals

def cluster_signals_by_riu(signals: list[dict]) -> dict[str, list]:
    """Group gap signals by RIU ID."""
    clusters = {}
    for signal in signals:
        riu_id = signal.get("riu_id", "unknown")
        clusters.setdefault(riu_id, []).append(signal)
    return clusters

def emit_gap_report(clusters: dict[str, list]) -> dict:
    """Build gap report — RIUs with 3+ signals need attention."""
    report = {
        "generated_at": datetime.now(timezone.utc).isoformat(),
        "high_priority": [],  # 5+ signals
        "medium_priority": [],  # 3-4 signals
        "low_priority": [],  # 1-2 signals
    }
    for riu_id, signals in sorted(clusters.items(), key=lambda x: -len(x[1])):
        entry = {"riu_id": riu_id, "signal_count": len(signals), 
                 "sample_queries": [s.get("query", "")[:100] for s in signals[:3]]}
        if len(signals) >= 5:
            report["high_priority"].append(entry)
        elif len(signals) >= 3:
            report["medium_priority"].append(entry)
        else:
            report["low_priority"].append(entry)
    return report

def write_gap_report(report: dict) -> Path:
    """Write gap report to proposals dir."""
    PROPOSALS_DIR.mkdir(parents=True, exist_ok=True)
    date_str = datetime.now(timezone.utc).strftime("%Y-%m-%d")
    out_path = PROPOSALS_DIR / f"gap_report_{date_str}.json"
    with open(out_path, "w") as f:
        json.dump(report, f, indent=2)
    return out_path

if __name__ == "__main__":
    signals = read_gap_signals(days=7)
    clusters = cluster_signals_by_riu(signals)
    report = emit_gap_report(clusters)
    out = write_gap_report(report)
    print(f"Gap report: {out}")
    print(f"High priority RIUs: {len(report['high_priority'])}")
    print(f"Medium priority RIUs: {len(report['medium_priority'])}")
```

This is ~80 lines. It reads from `session_log.ndjson` (which already exists), clusters by RIU, and writes a gap report. Running this daily (via cron or `health_check.py`) closes the loop from EXTRACT → gap detection → human-reviewable report.

## Synthesis: This Is the OS Claim, Made Architectural

The North Star Vision says: "Palette is the OS for professional judgment." That claim requires the routing loop to be obligatory — not a convention, not a best practice, but a structural guarantee.

Here is the structural guarantee the sprint above creates:

1. **A query that can't be classified can't be routed.** The pipeline aborts, the gap is logged. No silent degradation.

2. **A message that claims to request execution without a taxonomy classification will be rejected by the bus.** The infrastructure enforces it. Not the caller's discipline.

3. **Every external search is formulated from taxonomy state.** The system knows what it already has before asking for what it needs. The gap between "what Palette knows" and "what Perplexity finds" is explicit and structured.

4. **Every gap signal accumulates in a reviewable queue.** The system tells humans what it doesn't know. The humans decide whether to teach it.

5. **Self-improvement is human-reviewed at the taxonomy level, automated at the skill level.** New RIU nodes require approval. Skill maturity promotes automatically. The tier hierarchy prevents quality degradation.

This is the kernel analogy completed: user-space (agents) cannot access hardware (knowledge, execution) without going through the kernel (taxonomy + broker). The kernel enforces the contract. The contract makes the system trustable.

> "I design the smallest system that can be trusted in production."

The smallest system that can be trusted is one where the governance layer cannot be bypassed — not because it's inconvenient to bypass, but because bypassing it produces a structured error that cannot be silently ignored. That system exists in the current repo. The sprint above wires the last three enforcement points.
